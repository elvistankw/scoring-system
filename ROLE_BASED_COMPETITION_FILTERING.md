# 基于角色的比赛过滤功能实现

## 功能描述
确保评审仪表板只显示活跃和即将开始的比赛，而管理员可以看到所有状态的比赛，包括已结束的比赛。

## 实现方案

### 1. 后端API修改 ✅

#### 修改比赛控制器 (`backend/controllers/competitions.controller.js`)
在 `getAllCompetitions` 函数中添加基于用户角色的过滤逻辑：

```javascript
// 基于角色的过滤：评审只能看到活跃和即将开始的比赛
if (userRole === 'judge') {
  query += ` AND status IN ('active', 'upcoming')`;
}

// 如果评审尝试过滤已完成的比赛，返回空结果
if (userRole === 'judge' && status === 'completed') {
  return res.status(200).json({
    status: 'success',
    cached: false,
    data: { competitions: [] }
  });
}
```

#### 缓存键优化
更新缓存键以包含用户角色，确保不同角色的用户获得正确的缓存数据：

```javascript
const filterKey = `${status || 'all'}:${region || 'all'}:${type || 'all'}:${userRole || 'guest'}`;
```

### 2. 前端组件修改 ✅

#### 评审比赛选择器 (`components/judge/competition-selector.tsx`)
- 移除"已完成"状态选项
- 默认显示"进行中"的比赛
- 只提供"全部状态"、"进行中"、"即将开始"三个选项

```tsx
<select>
  <option value="all">全部状态</option>
  <option value="active">进行中</option>
  <option value="upcoming">即将开始</option>
  {/* 移除了 completed 选项 */}
</select>
```

#### 管理员比赛列表保持不变
管理员的比赛列表 (`components/admin/competition-list.tsx`) 继续显示所有状态：
- ✅ 即将开始 (upcoming)
- ✅ 进行中 (active) 
- ✅ 已结束 (completed)

## 技术实现细节

### 1. 后端安全控制
- **JWT认证**：所有比赛API都需要有效的JWT令牌
- **角色提取**：从JWT payload中提取用户角色 (`req.user.role`)
- **SQL过滤**：在数据库查询级别应用角色过滤
- **缓存隔离**：不同角色使用不同的缓存键

### 2. 前端用户体验
- **无感知过滤**：评审用户看不到已结束比赛的选项
- **清晰界面**：状态选择器只显示相关选项
- **一致性**：所有评审相关页面都遵循相同的过滤规则

### 3. 性能优化
- **缓存策略**：基于角色的缓存键确保正确的数据隔离
- **查询优化**：在数据库级别过滤，减少数据传输
- **前端缓存**：SWR缓存机制提高响应速度
